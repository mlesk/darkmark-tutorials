Lesson 0003: Mining Deep Dive — Consensus, Nonces & Security

⏱ ~20 minutes 📅 2026-06-10 🎯 Supplementary • Mining
Goal: Answer three deep questions about Bitcoin mining: (1) How does the network agree on which block to mine, without a central coordinator? (2) What does "solving the nonce" actually mean — explained so clearly a 10-year-old would get it. (3) What stops miners from cheating, and why can't a rich attacker just take over the network?

Part 1: Who Decides Which Block to Mine?

This is one of the most important questions in Bitcoin. If thousands of miners around the world are all hashing, who tells them which block to work on? Who picks the transactions? Who coordinates the race?

The answer: nobody. There is no coordinator. No vote. No round-robin. No leader election. Every miner independently builds their own candidate block, and the network converges after the fact — by everyone switching to build on whichever valid block wins the race.

The Block Template: Built Locally by Each Miner

Every mining node runs bitcoind (or equivalent software) and maintains its own mempool. When it's time to build a candidate block, the miner (or mining pool operator) does this locally:

  1. Pick the chain tip: Identify the block at the tip of the valid chain with the most accumulated proof-of-work. This is the "previous block" your new block will build on.
  2. Build the coinbase transaction: Create the special first transaction that pays your own address the block subsidy (3.125 BTC) plus all fees from the transactions you include. This is why miners don't need to be told who to pay — they pay themselves.
  3. Select transactions from your mempool: Sort by feerate (highest sat/vB first), pack them in until you hit the 4 million weight unit limit.
  4. Compute the Merkle root — a single 32-byte hash fingerprinting every transaction in the block.
  5. Assemble the 80-byte block header (version, previous block hash, Merkle root, timestamp, difficulty target, nonce = 0) and start hashing.
Every miner builds a different block template: 
Miner A in Texas          Miner B in Kazakhstan        Miner C in Japan
┌─────────────────┐      ┌─────────────────┐        ┌─────────────────┐
│ Coinbase → A's   │      │ Coinbase → B's   │        │ Coinbase → C's   │
│   address        │      │   address        │        │   address        │
│                 │      │                 │        │                 │
│ Txs: A's mempool │      │ Txs: B's mempool │        │ Txs: C's mempool │
│ (slightly diff)  │      │ (slightly diff)  │        │ (slightly diff)  │
│                 │      │                 │        │                 │
│ Merkle root:     │      │ Merkle root:     │        │ Merkle root:     │
│  a1b2c3d4...     │      │  e5f6g7h8...     │        │  i9j0k1l2...     │
└─────────────────┘      └─────────────────┘        └─────────────────┘
      │                         │                           │
      ▼                         ▼                           ▼
  Hash away!                Hash away!                  Hash away!
🔑 Key insight: Because every miner's coinbase pays a different address, every miner's Merkle root is different. This means every miner is hashing a different block header. They are not racing to solve the same puzzle — they are each solving their own puzzle, and the first one to find a valid solution wins the round.

How the Network Converges: The Emergent Consensus

There is no "agreement" before the block is found. The agreement happens after:

⛏ Miner finds valid block
→
📡 Broadcasts to peers
→
✅ Every node validates independently
→
🔗 All miners switch to building on top of it
→
🔄 Next round begins

The longest valid chain (more precisely: the chain with the most accumulated proof-of-work) is the one everyone follows. If you try to mine on an old block, you're racing not just against one block, but against the entire honest network that's already built on a newer tip. You would need to outrun the combined hashrate of everyone else — which is why it's rational to always build on the tip.

Source: Bitcoin Developer Guide — Block Prototypes; Satoshi Nakamoto — Bitcoin Whitepaper, §3–4

Part 2: Solving the Nonce — A 10-Year-Old's Guide

Imagine a Magic Dice Game

Let's play a game. I'm thinking of a giant 256-sided die (well, actually, a number so big it has 78 digits). Your job is to roll the die over and over until you get a number smaller than 5.

That's hard! On average, you'd need to roll about 51 times to get a number under 5 (since only 1, 2, 3, and 4 count out of 256 faces). The more times you roll per second, the faster you'll find a winning roll.

Now imagine I make it harder: you need a number smaller than 1. Now the only winning roll is 0 — and you'll need to roll about 256 times on average. Even harder: what if you needed to roll a number smaller than 0.00000000000000000001? You'd be rolling for a very, very long time.

This is exactly what Bitcoin mining is. But instead of a die, miners use a mathematical function called SHA-256 (applied twice). Instead of rolling, they feed the block header into SHA-256 and get a 256-bit number out. If that number is below the target, they win. If not, they change one number in the header — the nonce — and try again.

What Is SHA-256? The Fingerprint Machine

SHA-256 is a cryptographic hash function. Think of it as a fingerprint machine:

  • You put anything in — a single word, a whole book, a block header. It always gives you a 256-bit fingerprint (64 hex characters).
  • Change one tiny thing in the input — even a single bit — and the fingerprint changes completely and unpredictably.
  • You cannot work backwards from the fingerprint to find the input. The only way to find an input that produces a specific fingerprint is to guess and check.
SHA-256 in action (simplified): 
SHA256("Hello")   = 185f8db3...a52f6f49  (starts with 1 — not enough zeros)
SHA256("Hello!")  = 336a6101...2b5a67b1  (completely different!)
SHA256("Hello!!") = 0a5f66c1...8829db4e  (still no good)

Bitcoin uses DOUBLE SHA-256:
SHA256(SHA256(block_header)) must be < target

The Nonce: A 4-Billion-Combination Lock

The nonce is a 32-bit number (0 to 4,294,967,295) in the block header. The miner's ASIC tries: 

Try nonce = 0:      SHA256²(header with nonce=0)      → 7a3f...  Too big ✗
Try nonce = 1:      SHA256²(header with nonce=1)      → e2c1...  Too big ✗
Try nonce = 2:      SHA256²(header with nonce=2)      → 9d4b...  Too big ✗
...
Try nonce = 847,203,661: SHA256²(header with nonce=...) → 0000... Below target ✓✓✓
                    └── BLOCK SOLVED! ──┘

Modern ASICs try hundreds of trillions of nonces per second. But 4 billion nonces isn't enough when the difficulty is high — so miners also tweak the extra nonce in the coinbase transaction, which changes the Merkle root, giving them a fresh 4 billion nonces to try. They repeat this continuously until someone, somewhere, finds a valid hash.

Why "Leading Zeros"?

You may have heard "miners look for a hash starting with many zeros." This is because the target is a very small number. In hexadecimal, small numbers start with zeros. The more leading zeros required, the harder the problem.

Easy target: 0 0 a 3 f 1 b c … …
Hard target: 0 0 0 0 0 0 0 0 0 3 a 1 f

At the current network difficulty (~112 trillion), the target is so small that a valid block hash needs roughly 19 leading zeros in hex. The entire network — around 800 exahashes per second — takes about 10 minutes on average to find one.

Putting It All Together for a 10-Year-Old

Imagine every miner in the world has a magic bingo machine. Each machine shakes a giant bucket of numbered balls and spills them out. If the numbers that come out are all very small numbers (mostly zeros), the miner shouts "BINGO!" and wins a prize — 3.125 brand-new bitcoins plus all the transaction tips.

The machines shake incredibly fast — trillions of times per second. The whole world's machines shake together, but they all have slightly different buckets (because each miner's prize address is different, which changes the balls inside). The first machine to get a "bingo" wins, and everyone immediately resets and starts shaking for the next round.

The game automatically adjusts: if bingos start coming too fast (faster than every 10 minutes), the game makes the target smaller — you need even smaller numbers. If bingos are too slow, it makes the target bigger. This happens every 2,016 rounds (~2 weeks).

That's Bitcoin mining. Trillions of magic bingo shakes per second, worldwide, with a new winner every ~10 minutes.

Part 3: Why Miners Can't Cheat — The Security Model

Bitcoin's security doesn't come from trust, police, or a central authority. It comes from game theory and economic incentives.

The Core Incentive: Honest Mining Is Profitable

A miner who plays by the rules earns:

Revenue per block = 3.125 BTC (subsidy) + fees from all included transactions

≈ $300,000+ per block at current prices

Mining costs real money — electricity, hardware depreciation, cooling, facility costs. A cheater who gets caught (or whose invalid block is rejected by the network) earns zero for all that work. The economic incentive is overwhelmingly to mine honestly.

What Can a Miner Actually Try to Cheat At?

✅ Things miners CAN do

  • Choose which transactions to include (or exclude)
  • Try to mine empty blocks
  • Attempt to reorganise the chain
  • Try to double-spend their own coins

❌ Things miners CANNOT do

  • Steal bitcoin from someone else's address (requires forging a signature — mathematically impossible)
  • Create bitcoin out of thin air beyond the subsidy (nodes reject it)
  • Change the 21 million supply cap (consensus rule)
  • Spend someone else's UTXOs (can't produce their signature)
🔑 Critical distinction: A 51% attacker can double-spend their own coins (by broadcasting one transaction, mining a secret chain that spends the same coins elsewhere, then revealing the longer secret chain). They cannot steal your coins. Your private key is still your private key. The attack is economically disruptive but not theft of others' funds.

The 51% Attack: Why It's So Hard

A 51% attack (more accurately: a majority-hashrate attack) means an attacker controls more than half of the total network hashrate. With majority hashrate, they can:

  1. Double-spend: Send bitcoin to an exchange, get fiat withdrawn, then mine a longer chain where that transaction never happened and recover the bitcoin.
  2. Censor transactions: Refuse to include certain transactions in any block they mine.
  3. Block other miners: Always build on their own blocks, ignoring (orphaning) blocks from honest miners.

Why No One Has Successfully Done It (Sustainably)

💰

Astronomical Cost

To control 51% of the network (~400 EH/s), you'd need roughly $20–40 billion in ASIC hardware, plus ongoing electricity costs of ~$10–15 million per day. The attack burns money continuously while the honest chain keeps advancing.

🏭

Hardware Scarcity

You can't just "buy" 51% of the hashrate. ASICs are manufactured by a handful of companies (mostly Bitmain and MicroBT) with finite production capacity and long lead times. You'd need to outbid the entire existing mining industry for years of production.

⚡

Energy Constraints

400 EH/s consumes roughly 12–15 gigawatts — comparable to the entire electricity consumption of a mid-sized country. Finding that much power, getting grid connections, and building facilities takes years.

📉

Self-Defeating

A successful 51% attack would destroy confidence in Bitcoin, crashing the price. The attacker's hardware (purpose-built for SHA-256 mining) would become near-worthless. The billions spent on the attack would vaporise. You'd spend $30B to maybe steal $5B — and destroy the asset in the process.

🌍

Geographic Distribution

Mining is spread across the globe — United States, Kazakhstan, Russia, Canada, and dozens of other countries. No single jurisdiction can compel miners to collude. Pool operators can be pressured, but miners can switch pools in minutes.

👁️

Detectable & Responsive

A 51% attack is immediately visible (large reorgs, censored transactions). The community can respond: exchanges can raise confirmation requirements, developers can propose checkpointing or PoW changes, and honest miners have every incentive to coordinate a counter-response. The attacker can't do it secretly.

What About Mining Pools?

Mining pools introduce a subtlety. A single pool controlling >50% of hashrate is a theoretical concern — and it has come close (GHash.io briefly exceeded 50% in 2014). But pool hashrate is not the same as attacker hashrate:

  • Pool miners are mercenaries: If a pool operator tries to attack the network, individual miners can — and will — switch to another pool within minutes. The pool's hashrate evaporates.
  • Stratum V2 (in deployment): Allows individual miners to construct their own block templates rather than accepting the pool's template. This decentralises transaction selection even within pools.
  • Economic self-interest: Pool operators earn fees from honest mining. An attack kills their business model. The most profitable strategy is to mine honestly.

Blockchain Rewrites: The Real Limits of 51%

Even with 51% hashrate, you cannot rewrite arbitrarily far back. The further back you go, the more blocks you need to re-mine, and the honest chain keeps advancing while you try to catch up. The probability of catching up from k blocks behind decreases exponentially with k. This is why 6 confirmations is the standard — after 6 blocks, the probability of a successful reorg by an attacker with even 40% of hashrate is effectively zero.

What 51% Does NOT Enable

A common misconception: "If someone controls 51% of mining, they can steal everyone's bitcoin." This is false.

  • Cannot forge signatures: Spending from an address requires a valid ECDSA (or Schnorr) signature. Mining hashrate doesn't break cryptography. SHA-256 hashrate has nothing to do with ECDSA key security.
  • Cannot change consensus rules: Nodes independently validate every block. A block that creates excess bitcoin, spends non-existent UTXOs, or breaks any consensus rule is rejected by every full node — regardless of how much hashrate produced it.
  • Cannot reverse ancient transactions: The attacker can only reorganise the chain from the point they start their attack. After sufficient confirmations, the work required exceeds any practical attacker's resources.

Key Takeaways

  1. There is no coordinator. Every miner independently builds a candidate block with their own coinbase address. The network converges after a valid block is found — by everyone switching to build on the winning tip.
  2. Mining = guessing. SHA-256 is a one-way fingerprint machine. The only way to find a hash below the target is to try nonces trillions of times per second until you get lucky. The "nonce" is just a counter you keep incrementing.
  3. The coinbase makes every miner's puzzle different. Because each miner pays themselves at a different address, their Merkle roots differ, and they hash different headers. This is what prevents miners from competing on identical work.
  4. Honest mining is the most profitable strategy. Block rewards are worth hundreds of thousands of dollars. Cheating risks earning nothing for enormous electricity expenditure.
  5. A 51% attack is devastatingly expensive and limited in scope. It would cost billions in hardware and millions per day in electricity — and it cannot steal coins, forge signatures, or break consensus rules. It's economically irrational.
  6. The network's distribution is its defence. Thousands of independent miners across dozens of countries, pool-switching flexibility, hardware scarcity, and energy constraints make collusion practically infeasible.
